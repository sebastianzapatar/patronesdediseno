Principios SOLID

Son la base del diseño de software limpio y escalable. Todo patrón de diseño existe gracias a ellos:

S - Single Responsibility (Responsabilidad Única)
O - Open/Closed (Abierto/Cerrado)
L - Liskov Substitution (Sustitución de Liskov)
I - Interface Segregation (Segregación de Interfaces)
D - Dependency Inversion (Inversión de Dependencias)

S - Single Responsibility

Una clase debe tener una, y solo una, razón para cambiar.


// MAL: Mezcla lógica de negocio con base de datos
class Empleado {
    public void calcularSueldo() { /* ... */ }
    public void guardarEnBaseDeDatos() { /* ... */ }
}

// BIEN: Separación estricta de responsabilidades
class Empleado {
    public void calcularSueldo() { /* ... */ }
}
class EmpleadoRepository {
    public void guardarEnBaseDeDatos(Empleado e) { /* ... */ }
}

O - Open/Closed

Abierto para extender funcionalidad, cerrado para modificar código.


// Interfaz base (Cerrada a modificación)
interface Descuento {
    double aplicar(double precio);
}

// Nuevas extensiones (Abiertas) sin tocar el código existente
class DescuentoVIP implements Descuento {
    public double aplicar(double precio) { return precio * 0.8; }
}
class DescuentoEstudiante implements Descuento {
    public double aplicar(double precio) { return precio * 0.5; }
}

L - Liskov Substitution

Toda clase derivada debe sustituir a su base sin romper el sistema.


abstract class Ave { 
    abstract void comer(); 
}
abstract class AveVoladora extends Ave { 
    abstract void volar(); 
}

class Loro extends AveVoladora { /* Sabe volar y comer */ }

class Pinguino extends Ave { 
    // Solo sabe comer. Nunca se le pedirá volar,
    // evitando excepciones en tiempo de ejecución.
}

I - Interface Segregation

Múltiples interfaces pequeñas son mejores que una sola gigante.


// BIEN: Interfaces segregadas
interface Impresora { void imprimir(); }
interface Escaner { void escanear(); }

class MaquinaModerna implements Impresora, Escaner { /* ... */ }

// Una impresora vieja NO está forzada a implementar 'escanear()'
class ImpresoraBasica implements Impresora { 
    public void imprimir() { /* ... */ } 
}

D - Dependency Inversion

Depende de abstracciones (interfaces), no de concreciones.


interface BaseDatos { void guardar(String datos); }

class MySQL implements BaseDatos { /* ... */ }
class Mongo implements BaseDatos { /* ... */ }

// El negocio IGNORA si es MySQL o Mongo, depende de la abstracción
class RegistroNegocio {
    private BaseDatos db;
    
    public RegistroNegocio(BaseDatos db) {
        this.db = db; // Inyección de dependencia
    }
}

¿Qué son los Patrones?

Son soluciones habituales a problemas comunes en el diseño de software orientado a objetos. Cada patrón es como un plano o receta que se puede personalizar para resolver un problema particular en tu código.

No son código copiable: Son conceptos arquitectónicos estructurados. No puedes simplemente "copiar y pegar" un patrón, debes adaptar y analizar el concepto base para tu lenguaje y necesidad concreta.

Clasificación de Patrones

Creacionales

Cómo se construyen los objetos. Mecanismos seguros de instanciación.

Estructurales

Cómo se estructuran las clases, ensamblando arquitecturas mayores.

Comportamiento

Cómo interactúan entre sí, controlando flujo y responsabilidades.

Resiliencia

Cómo toleran fallos. Previenen caídas en cascada (Microservicios).

Patrones Creacionales

Mecanismos seguros y flexibles para la creación de objetos

Singleton CREACIONAL

Garantiza que una clase tenga una única instancia en todo el ciclo de vida del programa, a la vez que proporciona un punto de acceso global a dicha instancia.

  • Oculta el constructor haciéndolo private.
  • Proporciona un método estático como getInstancia().
  • Aplicación Real: Conexiones a Bases de Datos, Sistemas de Logging, Gestores de Configuración Centralizados.
Ilustración extraída de refactoring.guru

Factory Method CREACIONAL

Proporciona una interfaz para crear objetos en una superclase abstracta, mientras permite a las subclases alterar el tipo de objetos concretos que se crearán.

  • Delega la responsabilidad de instanciar a clases derivadas.
  • Desacopla el código cliente de las clases de inicialización pura.
  • Aplicación Real: Sistemas de Logística (Camión vs Barco), Gestores de Notificaciones (Email, SMS, Push).
Ilustración extraída de refactoring.guru

Builder CREACIONAL

Permite construir objetos complejos paso a paso. Nos permite producir distintos tipos y representaciones de un objeto empleando el mismo código de construcción base.

  • Evita la aberración de constructores gigantes (Telescoping Constructor Anti-pattern).
  • Permite tener un "Director" que orquesta el orden de los pasos.
  • Aplicación Real: Creación de Reportes Dinámicos, Construcción de Consultas de Base de Datos.
Ilustración extraída de refactoring.guru

Prototype CREACIONAL

Permite copiar objetos existentes sin hacer que el código dependa de sus clases concretas, usando el poderoso concepto de clonación.

  • Evita el uso repetitivo de new y los costosos tiempos de inicialización o llamadas a Base de Datos.
  • Requiere implementar Clonación Profunda (Deep Copy) si el objeto tiene referencias a arreglos o listas.
  • Aplicación Real: Sistemas de spawneo de enemigos en Videojuegos, Caché de plantillas de UI.

Patrones Estructurales

Ensamblaje de clases y objetos en arquitecturas mayores sin perder eficiencia

Adapter ESTRUCTURAL

Permite la colaboración fluida entre objetos con interfaces totalmente incompatibles. Actúa como un traductor entre dos mundos de código que no se entienden de forma nativa.

  • Envuelve ("wraps") a un objeto antiguo o tercero para esconder su complejidad.
  • Convierte formatos de datos automáticamente (Ej. Dólares a Centavos).
  • Aplicación Real: Integración de APIs de terceros obsoletas, Pasarelas de Pago heredadas (Legacy).
Ilustración extraída de refactoring.guru

Decorator ESTRUCTURAL

Permite añadir funcionalidades extra a objetos al vuelo (en tiempo de ejecución) colocando estos objetos dentro de clases encapsuladoras especiales.

  • Es una alternativa mucho más flexible y modular a la herencia pura.
  • Puedes apilar (stack) múltiples decoradores en capas estilo "Matryoshka".
  • Aplicación Real: Compresión o Encriptación dinámica al guardar archivos o enviar datos por la red.

Patrones de Comportamiento

Comunicación e interacción algorítmica y asignación de responsabilidades

Observer COMPORTAMIENTO

Define un poderoso mecanismo de suscripción (Publicador-Suscriptor) para notificar masivamente a varios objetos sobre cualquier evento que le suceda al objeto observado.

  • El publicador reacciona al instante pero ignora el contexto de quién lo observa.
  • Los observadores pueden entrar o salir de la lista de difusión en tiempo real.
  • Aplicación Real: Mercados Bursátiles y de Cripto, Eventos de Botones en UI.
Ilustración extraída de refactoring.guru

Strategy COMPORTAMIENTO

Permite definir una familia de algoritmos relacionados, encapsular cada uno de ellos y hacerlos estrictamente intercambiables. Extrae el "cómo" se hace.

  • Elimina enormes condicionales repetitivos (Anti-patrones de if/else encadenados).
  • Aísla la lógica matemática variable de las reglas inmutables de negocio.
  • Aplicación Real: Rutas GPS dinámicas (A pie, en auto), Cálculo algorítmico de envíos logísticos.

Patrones de Resiliencia

Mecanismos modernos para tolerar fallos en entornos distribuidos

Circuit Breaker RESILIENCIA

Previene que una aplicación intente ejecutar constantemente una operación remota que probablemente va a fallar.

  • Simula un interruptor eléctrico con estados: Cerrado, Abierto, Semi-Abierto.
  • Evita la sobrecarga de un servicio caído y los fallos en cascada.
  • Aplicación Real: Consumo de APIs inestables, arquitecturas de microservicios.
