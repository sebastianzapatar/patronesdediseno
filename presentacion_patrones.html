¿Qué son los Patrones?

Son soluciones habituales a problemas comunes en el diseño de software orientado a objetos. Cada patrón es como un plano o receta que se puede personalizar para resolver un problema particular en tu código.

No son código copiable: Son conceptos arquitectónicos estructurados. No puedes simplemente "copiar y pegar" un patrón, debes adaptar y analizar el concepto base para tu lenguaje y necesidad concreta.

Clasificación

Creacionales

Proporcionan mecanismos de creación de objetos seguros que incrementan la flexibilidad y la reutilización del código en el sistema.

Estructurales

Explican y estandarizan cómo ensamblar objetos y clases en estructuras más grandes sin sacrificar la eficiencia ni la fluidez del código.

Comportamiento

Se encargan específicamente de la comunicación efectiva, el control de flujo y la asignación de responsabilidades entre objetos.

Singleton CREACIONAL

Garantiza que una clase tenga una única instancia en todo el ciclo de vida del programa, a la vez que proporciona un punto de acceso global a dicha instancia.

  • Oculta el constructor haciéndolo private.
  • Proporciona un método estático como getInstancia().
  • Aplicación Real: Conexiones a Bases de Datos, Sistemas de Logging, Gestores de Configuración Centralizados.

Factory Method CREACIONAL

Proporciona una interfaz para crear objetos en una superclase abstracta, mientras permite a las subclases alterar el tipo de objetos concretos que se crearán.

  • Delega la responsabilidad de instanciar a clases derivadas.
  • Desacopla el código cliente de las clases de inicialización pura.
  • Aplicación Real: Sistemas de Logística (Camión vs Barco), Gestores de Notificaciones (Email, SMS, Push).

Builder CREACIONAL

Permite construir objetos complejos paso a paso. Nos permite producir distintos tipos y representaciones de un objeto empleando el mismo código de construcción base.

  • Evita la aberración de constructores gigantes (Telescoping Constructor Anti-pattern).
  • Permite tener un "Director" que orquesta el orden de los pasos.
  • Aplicación Real: Creación de Reportes Dinámicos, Construcción de Consultas de Base de Datos.

Adapter ESTRUCTURAL

Permite la colaboración fluida entre objetos con interfaces totalmente incompatibles. Actúa como un traductor entre dos mundos de código que no se entienden de forma nativa.

  • Envuelve ("wraps") a un objeto antiguo o tercero para esconder su complejidad.
  • Convierte formatos de datos automáticamente (Ej. Dólares a Centavos).
  • Aplicación Real: Integración de APIs de terceros obsoletas, Pasarelas de Pago heredadas (Legacy).

Decorator ESTRUCTURAL

Permite añadir funcionalidades extra a objetos al vuelo (en tiempo de ejecución) colocando estos objetos dentro de clases encapsuladoras especiales.

  • Es una alternativa mucho más flexible y modular a la herencia pura.
  • Puedes apilar (stack) múltiples decoradores en capas estilo "Matryoshka".
  • Aplicación Real: Compresión o Encriptación dinámica al guardar archivos o enviar datos por la red.
Ilustración extraída de refactoring.guru

Observer COMPORTAMIENTO

Define un poderoso mecanismo de suscripción (Publicador-Suscriptor) para notificar masivamente a varios objetos sobre cualquier evento que le suceda al objeto observado.

  • El publicador reacciona al instante pero ignora el contexto de quién lo observa.
  • Los observadores pueden entrar o salir de la lista de difusión en tiempo real.
  • Aplicación Real: Mercados Bursátiles y de Cripto, Eventos de Botones en UI.
Ilustración extraída de refactoring.guru

Strategy COMPORTAMIENTO

Permite definir una familia de algoritmos relacionados, encapsular cada uno de ellos y hacerlos estrictamente intercambiables. Extrae el "cómo" se hace.

  • Elimina enormes condicionales repetitivos (Anti-patrones de if/else encadenados).
  • Aísla la lógica matemática variable de las reglas inmutables de negocio.
  • Aplicación Real: Rutas GPS dinámicas (A pie, en auto), Cálculo algorítmico de envíos logísticos.

El Valor de los Patrones

Vocabulario Compartido

Conocer los patrones mejora radicalmente la comunicación de ingeniería en tu equipo de desarrollo. Decir "Aplica un Abstract Factory" es mil veces más eficiente que explicar toda la lógica manual de control e instanciación de clases relacionales.
